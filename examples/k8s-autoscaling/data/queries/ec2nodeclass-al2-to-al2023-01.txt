we just switched our EC2NodeClass from AL2 to AL2023 and now nodes launch but never join the cluster. they show up in the EC2 console as running but never appear in `kubectl get nodes`. here's our EC2NodeClass:

```yaml
apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: default
spec:
  role: KarpenterNodeRole-prod-cluster
  amiFamily: AL2023
  amiSelectorTerms:
    - alias: al2023@latest
  subnetSelectorTerms:
    - tags:
        karpenter.sh/discovery: prod-cluster
  securityGroupSelectorTerms:
    - tags:
        karpenter.sh/discovery: prod-cluster
  blockDeviceMappings:
    - deviceName: /dev/xvda
      ebs:
        volumeSize: 80Gi
        volumeType: gp3
        encrypted: true
        deleteOnTermination: true
  tags:
    environment: production
```

we recently migrated from aws-auth ConfigMap to EKS Access Entries. we have an access entry with type EC2_LINUX for the node role. this was working perfectly with AL2 nodes.

the nodes are launching, getting an IP, and the kubelet process starts. but it never successfully registers with the API server. cloud-init logs show the bootstrap completing without errors.

we're on EKS 1.31 with karpenter 1.0.6. the switch was literally just changing `amiFamily: AL2` to `amiFamily: AL2023` and updating the amiSelectorTerms. everything else stayed the same.

is there something different about how AL2023 nodes authenticate to the cluster? or is there an additional step needed for the Access Entry migration when switching AMI families?
